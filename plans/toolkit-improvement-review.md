# Toolkit 고도화 및 버그 픽스 리뷰 (진행 중)

## 현재까지의 진행 상황
1. **CopyableText.jsx 하이브리드 정규식 적용**: 과거 데이터 포맷 `('영문명')`과 신규 포맷 `[@영문명@]` 모두를 지원하여 하위 호환성을 보장하도록 개선. (성공)
2. **ToolkitTab.jsx 아키텍처 개선 (SSOT)**: 툴킷 탭 내부의 독립적인 14일 갱신 및 DB 접근 로직을 모두 제거하고, 오직 `PlaceWikiDetailsView.jsx` 를 통해 데이터를 동기화하고 업데이트를 위임하도록 구조 변경. (성공)
3. **무한 루프(버튼 작동 불가) 해결**: 
   - 툴킷 탭에 "AI 툴킷 생성하기" 버튼을 눌렀을 때 `forceUpdate: true` 플래그를 누락하던 문제를 수정.
   - 엣지 펑션 내부의 `Defensor (NO_CHANGES)` 로직이 `forceUpdate: true`일 때는 발동하지 않도록 프롬프트를 수정하여 무조건 텍스트를 재생성하도록 강제.

## 발생한 문제와 원인 분석
로그 분석 결과, **"Gemini did not return valid JSON"** 오류가 백엔드(Edge Function)에서 발생하고 있음을 확인했습니다.

### 원인
1. Edge Function은 제미나이에게 `{ "status": "UPDATED", "markdown": "..." }` 형태의 단일 JSON 객체를 반환하라고 요청합니다.
2. 하지만 제미나이가 생성한 텍스트 안에 줄바꿈(`\n`), 쌍따옴표(`"`) 등이 완벽하게 이스케이프(Escape) 되지 않았거나, 제미나이가 아예 JSON 형식을 무시하고 일반 텍스트로 대답했을 가능성이 높습니다.
3. 이로 인해 `JSON.parse(cleanText)` 가 실패하며 Edge Function이 에러를 뿜고 강제 종료됩니다.
4. 강제 종료 시, DB 무한 로딩을 방지하기 위해 기존에 있던 (툴킷 포맷이 없는) 옛날 데이터로 DB를 롤백시킵니다.
5. 프론트엔드는 롤백된 옛날 데이터를 받고 파싱(`---[TOOLKIT_START]---` 없음)에 실패하여 또다시 "AI 툴킷 생성" 버튼을 화면에 띄웁니다.
6. 만약 이전 호출에서 에러 복구가 제대로 안 되어 `[[LOADING]]` 텍스트 자체가 DB에 남았다면 100% 무한 로딩 화면에 갇히게 됩니다. (이를 대비해 100% 도달 시 "강제 다시 시도" 구명조끼 버튼을 프론트에 추가해 두었음).

## 다음 세션에서 해결해야 할 작업 (Next Steps)
1. **Edge Function (update-place-wiki/index.ts) 파싱 로직 개선**:
   - 제미나이가 응답하는 데이터가 엄격한 JSON이 아니라 마크다운 텍스트 덩어리로 왔을 경우에도 `markdown` 필드로 강제 인식하여 추출하도록 예외 처리를 추가합니다.
   - 굳이 JSON 형식을 요구하지 않고 제미나이가 순수 마크다운(Text)만 뱉어내도록 시스템 프롬프트를 간소화하는 방법도 고려할 수 있습니다. (현재 JSON 구조가 필요한 유일한 이유가 `status: "NO_CHANGES"` 때문인데, `forceUpdate` 시에는 이조차 필요 없기 때문입니다.)
2. 프론트엔드의 구명조끼 버튼(100% 강제 다시 시도)이 제대로 동작하여 복구가 되는지 확인합니다.
