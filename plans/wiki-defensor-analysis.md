# 디펜서(Defensor) 통합 및 자동 갱신(Lazy Update) 오류 원인 분석

질문하신 두 가지 사항에 대한 핵심 진단 결과입니다.

---

## 1. 디펜서(Defensor) 모델 폐기 및 제미나이 2.5 Pro 통합 여부

- **답변**: 네, 맞습니다. 과거의 '디펜서(Flash-Lite) 판단 모델 -> 통과 시 본 모델(Pro) 글쓰기' 라는 2-Step 구조는 비용과 시간 최적화를 위해 **제미나이 2.5 Pro 하나로 통합(1-Step) 처리**되도록 변경되었습니다.
- 단일 모델(Pro)에게 시스템 프롬프트로 "이전 정보가 있다면 전쟁/자연재해 같은 중대 변동이 없는 한 `{ "status": "NO_CHANGES" }`만 반환해라" 라고 지시하여 디펜서의 역할까지 겸임하도록 설계되어 있습니다.

---

## 2. 왜 14일 갱신 시 "포맷이 붕괴된 새로운 글"을 뱉어냈는가? (치명적 오류 원인)

이번 아조레스 제도 14일 경과 갱신 시, 내용이 전혀 바뀔 이유가 없는데도(NO_CHANGES) 불구하고 **AI가 자기 마음대로 기존 정보를 무시하고 블로그 말투의 새로운 글을 창작(Overwrite)**해버린 치명적인 원인을 코드에서 발견했습니다!

### 💥 원인: System Prompt와 User Prompt의 "모순(충돌)"
현재 백엔드(`update-place-wiki/index.ts`) 코드를 보면,
- **System Prompt**에서는: "치명적인 변동이 없으면 절대 다른 내용 쓰지 말고 무조건 `{ "status": "NO_CHANGES" }`만 반환해!" 라고 지시합니다.
- 그런데 **User Prompt**(바로 다음 문장)에서는: "반드시 다음의 두 필드를 포함하는 JSON으로 응답해: `{ "status": "UPDATED", "markdown": "위키 본문..." }`" 이라고 **강제**하고 있습니다.

이 두 지시가 정면으로 충돌하면서, AI(제미나이)는 딜레마에 빠졌고 결국 User Prompt의 **"무조건 UPDATED 포맷으로 글을 써서 내놔"라는 명령에 굴복**하여, 변동 사항이 없는데도 억지로 새로운 블로그 형태의 글을 지어내서 덮어씌워버린 것입니다.
이 과정에서 기존의 4가지 섹션(🌟 1분 요약 등) 포맷 지시를 까먹거나 융합해버려, 아조레스 제도의 결과물처럼 엉망진창인 포맷이 탄생했습니다.

---

## 3. 해결 및 개선 기획안

이 모순된 논리를 바로잡고 14일 갱신의 무결성을 보장하기 위해 코드를 전면 수정하겠습니다.

### A. 조건부 출력 지시 (프롬프트 모순 해결)
User Prompt의 "무조건 UPDATED를 내놔"라는 문구를 삭제하고, 다음과 같이 조건을 명확히 합칩니다.
> "앞서 지시한 기준에 따라, 1) 기존 정보가 아예 없거나, 2) 중대한 변동 사항이 감지되어 새롭게 전체 마크다운을 갱신해야 할 경우에만 `{ "status": "UPDATED", "markdown": "..." }` 구조로 작성해. 만약 갱신이 불필요하면 `{ "status": "NO_CHANGES" }` 로만 대답해."

### B. "갱신/최초 생성" 상관없는 절대 포맷 강제(Double-lock)
AI가 어쩔 수 없이 새 글을 쓴다 하더라도, 절대 형식이 망가지지 않도록 User Prompt 쪽에 다음을 한 번 더 못 박습니다.
> "새로 글을 쓸 경우:
> 1. 제공된 4가지 지정 카테고리(`🌟 1분 요약`, `🛂 입국/비용 & 이동 팁` 등) 제목의 텍스트와 이모지를 절대 임의로 바꾸지 말고 그대로 복사해서 단락 제목으로 사용해라.
> 2. 절대 마크다운 헤딩(#, ##) 기호를 쓰지 마라.
> 3. 베테랑 가이드의 친절하고 신뢰감 있는 '경어체(해요체/하십시오체)'를 유지하고 불필요한 이모지(😉 등) 남발을 금지하라."

이 수정이 적용되면, 앞으로 14일 경과 자동 갱신이 돌더라도 "전쟁/재난"이 없으면 무조건 단 1~2초 만에 `NO_CHANGES`로 튕겨내며 기존 데이터를 안전하게 보존(비용 절감)할 것이고, 진짜로 새로 써야 할 때도 절대 포맷과 톤앤매너가 망가지지 않게 됩니다!
