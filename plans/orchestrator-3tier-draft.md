# 오케스트레이터 3단 검토안 (컨트롤러 → 지휘자 → 워커2)

**상태**: 검토안 · **미적용** (2026-07-23)  
**공식**: [`orchestrator-method.md`](./orchestrator-method.md) **v2.1** (메인 + 워커2 · §4.2 후임 이관)  
**목적**: 컨트롤러 컨텍스트를 아끼며 다배치 SSOT(명소 hub 등)를 장시간 돌린다.  
**관련 검토**: R48–R61 실운 · R62–R69 큐 대기 · 일지 [`2026-07-23-project-log.md`](./2026-07-23-project-log.md)

> 이 문서는 **공식 방법이 아니다**. v2.1이 안정된 뒤, Task 중첩이 검증되면 v3 후보로 올린다.

---

## 1. 구조

```text
컨트롤러(메인 채팅 · 장수)
  └─ 지휘자(서브 1)
       ├─ 워커A (서브 2)
       └─ 워커B (서브 2)
```

- Cursor 중첩: 메인 → 1단 서브 → 2단 서브까지 가능. 워커는 더 깊게 서브 금지.
- 클라우드·로컬 모두 Task로 기동 가능. **동일 tip 동시 편집 금지**.

| 역할 | 하는 일 | 금지 |
|------|---------|------|
| **컨트롤러** | 다음 R(배치표 2개) 전달 · 지휘자 1명 기동 · 반환 요약으로 PASS/FAIL만 판단 · PASS면 다음 세대 · FAIL·무응답 시 재시도 1회 또는 사람 보고 · **큐 소진 시 사람이 보는 최종 보고** | tip/JSON 전문 Read · 워커 직접 기동 · 직렬 머지 · audit/스모크 재실행 · 일지 장문 |
| **지휘자** | 워커 정확히 2 기동 · 초안 수신 · tip 직렬 append(A→B) · audit+스모크 VERIFY · 큐 ✅ · 일지 2~5줄 · §3.3 문제 조치 | `main` 강제 push · SSOT 전면 rewrite · 컨트롤러처럼 장수 루프만 돌리며 tip 방치 |
| **워커** | 배정 1배치(관례 5 hub) 초안 JSON + 스모크 쿼리 요약만 반환 | tip append · push · 이관서 · 다른 워커 tip 건드림 |

## 2. 세대 루프

```text
컨트롤러: 큐에서 다음 R (A5+B5) 확정
    ↓
지휘자 1명 기동 (배치표만 전달, tip 전문 금지)
    ↓
지휘자: 워커2 병렬 초안 → tip 직렬 머지(A→B) → VERIFY → 큐/일지
    ↓
컨트롤러에 요약만 반환
    ↓
PASS → 다음 R로 지휘자 재기동
FAIL/무응답 → 재시도 1회 또는 사람 escalate
큐 소진 → 컨트롤러가 사람 채팅에 최종 보고 (hub/명소 건수 · audit 0 · 커밋 여부)
```

### 컨트롤러가 받는 요약 (이것만 · JSON 전문 금지)

- R번호 · tip 건수(hub/명소)
- VERIFY: audit issues / 스모크 PASS|FAIL
- 큐 상태: 해당 R ✅|⬜ · 스킵/예비 대체 1줄
- 다음 R 배치표 2개 (워커A5 · 워커B5)
- 문제 시: §3.3 조치 1줄 또는 escalate 사유

### 컨텍스트 절감

- 컨트롤러는 상황 파악 + 가동 여부만. 정상 PASS면 tip에 개입하지 않음.
- 지휘자·워커 중간 로그·초안 JSON을 컨트롤러 컨텍스트에 넣지 않음.
- 「거의 무한」은 요약을 얇게 유지할 때만 성립. 요약이 두꺼워지면 컨트롤러도 찬다.
- 감사·문제 조치의 실행은 지휘자. 컨트롤러는 FAIL/무응답 판단·재기동·**최종 사람 보고**만.

| | v2.1 (현재 공식) | 3단 검토안 |
|--|------------------|------------|
| 메인 | 오케스트레이터 (머지·VERIFY 포함) | 컨트롤러 (루프·최종 보고만) |

| 사람 보고 | **마지막 후임**이 큐 소진 시 보고 (첫 메인 침묵 가능) | **컨트롤러**가 항상 최종 보고 |

## 4. 왜 첫 메인(시작 지휘자)은 완료를 보고하지 않는가

관찰(R48–R61·이관 체인):

- 첫 스타트 메인은 **시작·1세대(또는 이관 Task 기동)** 만 하고 컨텍스트가 많이 남음.
- 이후 작업은 **후임 서브(오케스트레이터) 체인**이 이어 감.
- 큐가 다 비어도 **처음 채팅의 메인에게 “전부 끝남”이 자동으로 돌아오지 않음**.

### 원인 (후임 단계가 많아서만이 아님 — 설계상 끊김)

v2.1 §4.2는 이관을 **지휘권 이양**으로 정의한다.

1. 현 메인은 후임 Task를 띄운 뒤 **tip 추가·워커 추가를 중단**한다.  
2. 「사람에게 보고」는 **큐 소진·§3.3 E를 맡은 그 에이전트**(보통 **체인 끝 후임**) 몫이다.  
3. 후임은 **별도 Task/서브 컨텍스트**에서 일한다. 완료 알림이 상위(첫 메인)로 와도,  
   v2.1은 첫 메인에게 **요약을 모아 사람 채팅에 최종 보고하라**고 강제하지 않는다.  
4. 후임 단계가 늘수록:  
   - 사람이 보는 답변 위치가 **마지막 리프**로 이동하거나  
   - Task 실패·백그라운드 종료로 **아무 채팅에도 최종 보고가 안 남는** 경우가 늘다.  
5. 첫 메인의 컨텍스트 여유는 **의도된 결과**(일찍 이관해서 비움)이지,  
   “아직 지휘 중”이 아니다. **역할은 이미 넘긴 상태**라 완료 보고 의무가 없다.

```text
사람 ↔ 첫 메인(시작)
         │ §4.2 이관 후 tip/보고 의무 종료
         ▼
      후임1 → 후임2 → … → 후임N (큐 소진 시 여기만 사람 보고)
```

**정리**: 후임이 많아서 악화되는 면은 있으나, 근본은 **「이관 = 보고권도 함께 넘김」** 이라  
첫 메인이 여유 컨텍스트를 가진 채 **침묵하는 것이 v2.1 정상 동작**에 가깝다.

### 3단안이 이 구멍을 메우는 방식

- 사람 ↔ **컨트롤러만** 대화. 지휘자는 서브로 돌고 **요약만** 컨트롤러에 반환.
- 큐 소진·최종 VERIFY는 **컨트롤러가 사람 채팅에 보고** (컨텍스트도 비어 있어서 보고 가능).
- 첫 스타트 메인의 “여유 컨텍스트”를 **의도적으로 컨트롤러 장수**에 쓰는 설계.

### v2.1에서 당장 쓸 수 있는 완화 (3단 없이)

공식 변경 전이라도, 이관 프롬프트·세션 약속으로:

1. **체인 끝 후임**은 큐 소진 시 사람 보고 + 일지 2~5줄 (기존).  
2. **선택**: 첫 메인이 후임 Task를 `run_in_background: false`로 기다리거나,  
   완료 알림 후 **한 줄 최종 요약만** 사람 채팅에 중계 (tip 재작업 금지).  
3. 후임 N단이 길면 중계가 끊기기 쉬우므로, **세대마다 일지 상태 한 줄**이 사실상 SSOT 보고.
