# 제미나이 임베딩 모델(Gemini Embedding Model)을 활용한 장소 상세/지도 연동 방안 고찰

## 1. 개요 및 배경
현재 `gateo.kr` 서비스는 지구본(3D Globe) 시각화와 장소별 백과사전(Wiki)/리뷰(Logbook) 중심의 인터페이스를 제공하고 있습니다. 그러나 사이트 내에서 구글 지도, 네이버 지도 혹은 주변 맛집 정보 등으로 바로 넘어가는 명시적인 진입점(Entry Point)이 부족한 상태입니다. 

이를 해결하기 위해 **제미나이 텍스트/멀티모달 임베딩 모델(Text-Embedding-004 등) 및 LLM**을 활용하여 텍스트 및 이미지 데이터를 분석하고, 사용자가 탐색 중인 장소와 가장 관련된 지도 링크나 주변 명소/맛집 정보를 도출할 수 있는지 분석합니다.

## 2. 임베딩 모델의 역할과 한계

### 임베딩(Embedding)이란?
임베딩 모델은 텍스트(장소명, 리뷰 데이터)나 이미지를 고차원 벡터로 변환하여 두 데이터 간의 **유사도를 측정(Semantic Search)**하는 데 특화되어 있습니다. 

### 임베딩 모델의 한계점
임베딩 모델 자체는 좌표나 특정 "URL 링크"를 반환하는 생성형 모델이 아닙니다. 즉, "도쿄 타워 근처 맛집 링크를 찾아줘"라는 요청을 처리하기엔 부적합합니다. 이는 **텍스트 생성 모델(Gemini 2.5 Pro/Flash)**이나 **함수 호출(Function Calling/Tool use)** 기능과 결합해야 해결할 수 있습니다.

---

## 3. 제미나이를 활용한 지도/맛집 진입로 구현 시나리오

임베딩보다는 **제미나이 생성 모델(Gemini 2.5 Flash/Pro)의 Function Calling 및 장소 정보 추출 능력**을 활용하는 것이 더 적합합니다.

### A. 스마트 지도 링크 생성기 (Dynamic Map URL Generation)
제미나이 모델을 사용하여 현재 사용자가 보고 있는 `Place` 데이터(예: 이름, 좌표 등)를 기반으로 구글 지도나 애플 지도의 검색 URL을 동적으로 생성합니다.

- **방식**: 
  - 사용자가 보고 있는 위치 데이터(위도, 경도, 장소명)를 활용.
  - 구글 맵스의 Universal Cross-platform URL을 조합 (`https://www.google.com/maps/search/?api=1&query={위도},{경도}`)
  - 또는 제미나이에게 "이 장소를 구글 지도에서 검색할 수 있는 가장 정확한 검색 쿼리를 영문/현지어로 추출해줘"라고 요청한 후 해당 쿼리로 라우팅.

### B. "근처 맛집/명소" AI 큐레이션 및 진입점 (AI POI Curation)
- **방식**: 
  1. `PlaceCard` 컴포넌트 내에 `[📍 지도에서 보기]` 버튼과 함께 `[🍽️ 주변 로컬 맛집]` 토글 버튼 신설.
  2. 사용자가 버튼을 누르면 제미나이 API에 프롬프트 전송:
     > *"현재 위치는 [도쿄 시부야]입니다. 이 주변의 현지인들이 자주 가는 숨겨진 맛집 3곳을 추천하고, 각 맛집의 구글 맵 검색 URL을 함께 제공해주세요."*
  3. 제미나이가 JSON 형태로 맛집 이름, 간단한 설명, 검색 URL(Google Maps Search 쿼리)을 반환.
  4. 이를 UI 카드(가로 스크롤 혹은 리스트)로 렌더링.

### C. 임베딩(Embedding) 모델을 활용한 고도화 (RAG 연동)
만약 **기존 사용자들의 여행기(Logbook)** 데이터가 많이 쌓여있다면, 임베딩을 다음과 같이 활용할 수 있습니다.
- **방식**: 
  1. 사용자들의 여행기에 언급된 "식당", "카페" 리뷰 텍스트를 제미나이 임베딩 모델로 벡터화하여 Supabase의 `pgvector`에 저장.
  2. 다른 사용자가 특정 장소(예: "파리 에펠탑")를 열었을 때, "에펠탑"이라는 키워드 및 좌표를 임베딩하여 벡터 유사도 검색 수행.
  3. **"우리 사이트 사용자들이 추천한 근처 장소"** 라는 형태로 기존 리뷰 내 맛집/명소로의 내부 라우팅 혹은 외부 지도 라우팅을 제공.

## 4. 구현 제안 (Action Plan)

현재 사이트 구조에 가장 빠르고 효과적으로 통합할 수 있는 방법은 다음과 같습니다.

1. **장소 백과사전(Wiki) 또는 요약 뷰(Summary Mode) 내 버튼 추가**
   - **Google Maps 라우팅**: 별도 AI 호출 없이 현재 보유한 좌표(`lat`, `lng`)와 장소명(`name`)을 활용하여 단순 `a href`로 구글 맵 진입로 제공.
   - 예: `<a href={\`https://www.google.com/maps/search/?api=1&query=${lat},${lng}\`}>지도 열기</a>`

2. **AI 실용 정보(로컬 왓슨) 프롬프트 개편 (Gemini Flash 활용)**
   - 기존의 `update-place-wiki` Edge Function에서 제미나이를 호출할 때 프롬프트를 일부 수정.
   - "주변 추천 맛집이나 방문해볼 만한 스팟 2개를 추천하고, 각각을 검색할 수 있는 키워드를 제공해."
   - 반환받은 정보를 파싱하여 장소 카드 내 **"AI 추천 주변 명소"** 칩(Chip)이나 버튼으로 렌더링하고, 클릭 시 해당 키워드로 구글 맵스 검색 탭을 띄움.

## 5. 결론
임베딩 모델 단독으로는 외부 지도/맛집 링크로의 진입점을 만들 수 없지만, **Gemini 텍스트 모델의 구조화된 데이터 생성(JSON) 능력**이나 **기존 DB의 좌표 기반 URL 조합**을 사용하면 훌륭한 "외부 지도 및 맛집 진입로"를 만들 수 있습니다. 추후 유저 데이터가 방대해지면 임베딩/RAG를 결합하여 자체적인 '추천 맛집/명소 맵' 라우팅망을 구축하는 방향이 이상적입니다.
